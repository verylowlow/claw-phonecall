# 项目需求说明书：基于 Openclaw 的全软件 AI 电话 Agent (方案 A)

## 1. 项目概述
### 1.1 项目目标
通过纯软件手段实现 PC 与 Root 安卓手机之间的双向音频传输。利用 ADB 隧道传输原始 PCM 数据，在不增加物理外设的情况下完成 AI 拨打与接听电话。

### 1.2 核心挑战
* **音频路由接管：** 安卓系统默认不允许从 USB 接口向通话上行链路（Uplink）注入音频。
* **同步与缓冲：** 数字化传输过程中的网络抖动和缓冲区大小直接影响通话的实时感。

---

## 2. 核心架构设计：数字化音频桥接 (Digital Audio Bridge)

方案 A 不使用物理声卡，而是构建两条逻辑数据隧道。



### 2.1 捕获路径 (Phone -> PC)
* **方案：** 利用 `scrcpy` 2.0+ 的音频转发协议。
* **机制：** `scrcpy` 会在安卓端启动一个小的 `app_process` 代理，拦截系统的音频输出（Audio Capture API），通过 ADB 隧道将原始 **OPUS** 或 **RAW PCM** 数据推送到 PC。
* **PC 端接收：** 使用 Python 调用 `ffmpeg` 管道或 `scrcpy` 的输出流，将音频重定向至虚拟音频设备（如 VB-Audio Cable）供 ASR 处理。

### 2.2 注入路径 (PC -> Phone) — **本方案难点**
* **方案：** Root 级虚拟麦克风应用。
* **机制：** 在安卓端运行一个支持 Root 权限的辅助 App（如自定义开发的 `AudioInjector.apk` 或 **AudioRelay** 的 Root 模式）。
* **实现原理：** 1. PC 端将 TTS 生成的音频通过 Socket 发送到手机。
    2. 手机端 App 接收数据。
    3. 利用 Root 权限，通过 `AudioTrack` 类并指定 `STREAM_VOICE_CALL` 或 Hook 底层 `HAL` 层，将数据强制写入系统的通话录音输入口。

---

## 3. 功能模块详细说明

### 3.1 数字化控制 (ADB Control)
与方案 B 一致，通过 ADB 监控 `mCallState`：
* **监听：** `adb shell logcat -b radio`（获取更详细的通话信令）。
* **触发：** 监听到 `RINGING` 状态时，软件自动启动音频注入 Socket 连接。

### 3.2 动态延迟优化 (Latency Tuning)
* **零缓冲区策略：** 在 PC 端使用 `portaudio` 或 `sounddevice` 的低延迟模式，尽可能减小 ASR 的输入 Buffer。
* **采样率锁定：** 全链路锁定 **8,000Hz 或 16,000Hz (Mono)**。由于手机通话带宽本身较低，无需 44.1kHz，从而减少传输数据量。

### 3.3 拟人化策略 (AI Interaction)
* **VAD 介入：** Silero VAD 在 PC 端对来自 ADB 的音频流进行实时切片。
* **预录插入语：** 在 Openclaw Skill 中预加载“嗯”、“好的”等短 PCM 片段，一旦检测到 LLM 生成延迟超过设定阈值（如 $T > 1.2s$），直接向 Socket 发送填充音。

## 5. 方案 A 的优势与潜在坑点

### 5.1 优势 (Why Scheme A?)
* **极致简洁：** 一根 USB 线走天下，不需要任何声卡、隔离器或转接头。
* **音质一致性：** 数字化传输避免了模拟电路的底噪和电磁干扰（EMI）。
* **弹性扩展：** 可以通过软件轻松实现“一机多控”，在同一台 PC 上通过不同端口管理多部手机。

### 5.2 挑战 (Watch Out!)
* **回音消除 (AEC) 难题：** 由于是软件注入，手机硬件自带的 AEC（回音消除）可能无法识别这部分“伪装”的麦克风信号，导致对方听到极大的回声。
    * *对策：* 需要在 Openclaw 端集成软件 AEC 算法（如 WebRTC AEC）。
* **系统拦截：** 安卓的高版本系统对 `VOICE_COMMUNICATION` 通道的安全性保护极严，即便 Root 了，也可能出现注入失败的情况，需要针对特定机型做适配。

## 6. 赛马评估维度

当你后续测试时，请重点对比以下指标：
1.  **首句响应时间 (First Byte Latency)：** 从用户说完话到 Agent 开始出声的时间差。
2.  **回声反馈：** 对方是否能听到自己的声音。
3.  **系统负载：** 方案 A 涉及大量的音视频解压/编解码，对 PC CPU 的占用。
4.  **稳定性：** 连续通话 30 分钟后，音频流是否出现偏移（Sync Drift）或断续。

\## 一、 如何识别有电话拨入？



在 Openclaw 的 Skill 中，我们需要一个后台监听进程（Monitor）。这里有三种主流方案，按稳定性从高到低排列：



\### 1. 方案 A：ADB `dumpsys` 状态轮询 (最简单)

安卓系统会将通话状态实时记录在 `telephony.registry` 中。

\* \*\*命令：\*\* `adb shell dumpsys telephony.registry | grep mCallState`

\* \*\*状态码说明：\*\*

&#x20;   \* `mCallState=0`: 闲置 (IDLE)

&#x20;   \* `mCallState=1`: \*\*响铃 (RINGING)\*\* -> 此时有电话拨入！

&#x20;   \* `mCallState=2`: 通话中 (OFFHOOK)



\*\*逻辑实现：\*\* 在 Python 中启动一个死循环，每隔 500ms 执行一次该命令。一旦检测到状态从 `0` 变为 `1`，立即触发接听流程。



\### 2. 方案 B：ADB `logcat` 实时流监控 (延迟最低)

响铃时，系统日志会疯狂刷屏。我们可以通过监听特定关键词来捕获事件。

\* \*\*命令：\*\* `adb shell logcat -b radio | grep "RING"`

\* \*\*优点：\*\* 响应速度比轮询快，基本是毫秒级。

\* \*\*获取来电号码：\*\* 通过 `logcat` 还可以解析出 `incoming number`，让你的 Agent 在接听前就知道对方是谁（例如：“你好，张总”）。



\### 3. 方案 C：Tasker / MacroDroid 联动 (最稳定)

在手机上安装 Tasker（需要 Root 权限）。

\* \*\*配置：\*\* 设置一个触发器“来电时” -> 执行动作“发送 HTTP 请求”或“写入某个本地文件”。

\* \*\*优点：\*\* 这种方式最省电，且由于是系统原生广播触发，成功率接近 100%。

\## 二、 如何自动接听电话？



一旦识别到 `mCallState=1`，Openclaw 需要发送接听指令。



\### 1. 标准模拟按键指令

这是最通用的方法，模拟用户按下了耳机上的“接听”键：

\* \*\*命令：\*\* `adb shell input keyevent 5`

\* \*\*注意：\*\* 部分高版本安卓系统为了安全，可能禁用了 `input keyevent` 在锁屏状态下的接听功能。如果失效，需要先执行一次 `adb shell input keyevent 26`（唤醒屏幕）。



\### 2. Service Call 指令 (底层接听)

如果模拟按键不起作用，可以使用更底层的系统调用：

\* \*\*命令：\*\* `adb shell service call phone 1` (具体数字 `1` 可能随安卓版本变化，通常为 1-5 之间)。

\## 三、 完整的接听流程设计



当电话接入时，Openclaw 的 `PhoneAgentSkill` 应该执行以下序列：







1\.  \*\*检测响铃：\*\* 监控发现 `mCallState=1`。

2\.  \*\*提取信息：\*\* 从 `dumpsys` 或 `logcat` 提取来电号码（可选）。

3\.  \*\*执行接听：\*\* 发送 `adb shell input keyevent 5`。

4\.  \*\*确认通话建立：\*\* 循环检查 `mCallState` 直到变为 `2`。

5\.  \*\*启动 AI 管道：\*\*

&#x20;   \* \*\*第一句话策略：\*\* 为了降低延迟，接通后的 0.5 秒内，Agent 应立即通过 TTS 播放一句预设好的欢迎语（如：“喂，您好，请问哪位？”）。

&#x20;   \* \*\*加载 VAD/ASR：\*\* 在播放欢迎语的同时，后台启动音频采集，进入“听取”状态。



\---



\## 四、 特殊场景处理



\* \*\*拒绝来电：\*\* 如果来电号码在黑名单，执行 `adb shell input keyevent 6` (挂断)。

\* \*\*静音接听：\*\* 刚接通时，先保持 PC 端输出静音 200ms，避开接通瞬间的电流杂音。

\* \*\*通话状态同步：\*\* 如果对方挂断了，`mCallState` 会变回 `0`，此时 Skill 必须立即停止所有 ASR/TTS 进程，释放硬件占用，回到 `IDLE` 待命。



\---



\## 专家建议：



为了让 Openclaw 更像“人”，我建议在识别到响铃后，\*\*不要秒接\*\*。

让手机\*\*响铃 2-3 秒\*\*后再接听，这样对方的心理预期会更自然，同时也给了本地 ASR 和 TTS 模型预热（Warm-up）的时间。
